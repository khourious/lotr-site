O servidor LOTR (Learning Outbreak Trends in Real-Time) pertence ao Laboratório de Medicina e Saúde Pública de Precisão (MESP2) da Fiocruz Bahia, em Salvador. Ele fornece suporte a atividades científicas em bioinformática e análise de dados em saúde para usuários do próprio laboratório.

Seu uso é voltado ao processamento de dados de pesquisa em genômica, transcriptômica, metagenômica, biologia estrutural e epidemiologia molecular, com foco em aplicações para saúde pública. O acesso externo é proibido, exceto mediante autorização prévia.

Guia do usuário

Normas de acesso, segurança, uso de recursos, submissão de jobs e boas práticas para o servidor LOTR.

Finalidade de uso

O servidor deve ser utilizado exclusivamente para projetos acadêmicos e institucionais, processamento de dados científicos e desenvolvimento de pipelines computacionais.

É proibido o uso comercial, o processamento de dados privados de empresas e a execução de atividades não relacionadas à pesquisa.

Acesso e segurança

  • O acesso ao servidor é individual e intransferível.
  • É proibido compartilhar credenciais.
  • Senhas devem possuir no mínimo 8 caracteres, incluindo letras, números e caracteres especiais.

Primeiro acesso

Após a criação da conta, o usuário receberá por e-mail o username, o endereço de IP do servidor e a senha provisória. No primeiro acesso, será obrigatória a alteração da senha.

Alteração de senha

passwd

Ao digitar a senha, os caracteres não aparecem na tela. Isso é normal em sistemas Linux.

Após executar o comando, o sistema solicitará a senha atual, a nova senha e a confirmação da nova senha.

A nova senha deve atender aos requisitos mínimos de segurança. Caso a senha seja recusada, escolha uma combinação mais forte, evitando nomes, datas, palavras comuns ou senhas já utilizadas.

Em caso de perda da senha provisória ou problemas na alteração, entre em contato com a equipe responsável.

Acesso ao servidor

Antes de acessar o servidor, é necessário estar conectado à rede da Fiocruz. Fora da Fiocruz, o acesso via VPN é obrigatório.

  • Solicite autorização de acesso VPN junto à Fiocruz.
  • Instale e configure a VPN conforme o guia oficial: VPN - passo a passo da instalação.

O endereço de IP do servidor será enviado por e-mail junto com as informações de acesso.

ssh username@IP_DO_SERVIDOR

No Windows, pode-se utilizar PowerShell ou softwares como PuTTY.

Uso de alias para acesso

Para facilitar o acesso, o usuário pode configurar um alias SSH. Com isso, em vez de digitar o endereço completo do servidor a cada acesso, será possível usar um nome curto, como lotr.

No Linux, macOS ou Windows com OpenSSH, edite ou crie o arquivo ~/.ssh/config e adicione:

Host lotr
    HostName IP_DO_SERVIDOR
    User username

Depois disso, o acesso pode ser feito com:

ssh lotr

O mesmo alias também pode ser usado em transferências de arquivos, por exemplo:

scp arquivo.fastq.gz lotr:/data/username/projeto/

Transferência de arquivos

A transferência de arquivos entre o computador local e o servidor pode ser feita via scp, sempre utilizando o username e o endereço de IP recebidos por e-mail.

Para enviar um arquivo do computador local para o servidor:

scp arquivo.fastq.gz username@IP_DO_SERVIDOR:/data/username/projeto/

Para baixar um arquivo do servidor para o computador local:

scp username@IP_DO_SERVIDOR:/data/username/projeto/resultado.tsv .

Para transferir uma pasta inteira, use a opção -r:

scp -r minha_pasta username@IP_DO_SERVIDOR:/data/username/projeto/

Também é possível utilizar SFTP via terminal, que abre uma sessão interativa para navegar, enviar e baixar arquivos:

sftp username@IP_DO_SERVIDOR

Dentro da sessão SFTP, comandos como put, get, ls, cd e bye podem ser usados para transferir e navegar pelos arquivos.

A principal diferença é que scp é mais direto para copiar arquivos ou pastas em um único comando, enquanto sftp é mais conveniente para navegação interativa e múltiplas transferências na mesma sessão.

No Windows, além do PowerShell, o usuário pode utilizar programas com interface gráfica, como WinSCP, configurando o protocolo SFTP, o endereço de IP do servidor, o username e a senha recebidos por e-mail.

Criação e edição de scripts

Scripts são arquivos de texto contendo comandos que serão executados pelo sistema, como etapas de análise ou instruções de submissão de jobs. No servidor, eles podem ser criados e editados diretamente pelo terminal.

Para criar ou editar um arquivo com vim:

vim script.pbs

Também é comum encontrar o editor vi, usado da mesma forma:

vi script.pbs

Comandos básicos no vi/vim:

  • Pressione i para entrar no modo de edição.
  • Pressione Esc para sair do modo de edição.
  • Digite :w e pressione Enter para salvar.
  • Digite :q e pressione Enter para sair.
  • Digite :wq e pressione Enter para salvar e sair.
  • Digite :q! e pressione Enter para sair sem salvar.

Antes de submeter um script, revise caminhos, nomes de arquivos, ambiente computacional e recursos solicitados.

Uso de recursos computacionais com PBS

  • O uso dos recursos é gerenciado por sistema de filas PBS.
  • Análises pesadas não devem ser executadas diretamente no terminal.
  • Todo processamento intensivo deve ser submetido via fila.

As diretivas iniciadas por #PBS devem ser incluídas no início do script de submissão para informar ao sistema de filas quais recursos serão necessários.

No exemplo abaixo, o job solicita 16 CPUs e 64 GB de memória. Quando houver uso de GPU, a diretiva ngpus indica a quantidade de GPUs solicitadas.

#PBS -l select=1:ncpus=16:mem=64gb
#PBS -l mem=64gb
#PBS -l ngpus=1

O comando nvidia-smi não é uma diretiva PBS. Ele pode ser usado no terminal ou dentro do job para verificar se a GPU está disponível e acompanhar seu uso.

nvidia-smi

Verificação do sistema

Antes de iniciar análises ou submeter jobs, é recomendável verificar o estado geral do servidor e o espaço disponível. Esses comandos ajudam a avaliar carga, memória, CPUs e armazenamento.

htop
top
nproc
free -h
df -h
  • htop ou top: mostram processos em execução e uso de CPU/memória.
  • nproc: informa o número de CPUs disponíveis no sistema.
  • free -h: mostra o uso de memória RAM em formato legível.
  • df -h: mostra o uso de espaço em disco.

Para acompanhar jobs submetidos ao PBS, use os comandos abaixo:

qstat -u usuario
qstat
qstat -f JOB_ID
qdel JOB_ID
  • qstat -u usuario: lista os jobs do usuário.
  • qstat: lista os jobs na fila.
  • qstat -f JOB_ID: exibe informações detalhadas de um job específico.
  • qdel JOB_ID: cancela um job submetido.

Submissão de jobs

Jobs devem ser submetidos por meio de scripts PBS. O script abaixo define o nome do job, os recursos necessários, o tempo máximo de execução e os arquivos de saída e erro.

#!/bin/bash
#PBS -N meu_job
#PBS -l select=1:ncpus=16:mem=64gb
#PBS -l walltime=24:00:00
#PBS -o output.log
#PBS -e error.log

cd $PBS_O_WORKDIR

conda activate meu_env

python script.py
  • #PBS -N: define o nome do job.
  • #PBS -l select=1:ncpus=16:mem=64gb: solicita 16 CPUs e 64 GB de memória.
  • #PBS -l walltime=24:00:00: define o tempo máximo de execução.
  • #PBS -o e #PBS -e: definem os arquivos de saída padrão e erro.
  • cd $PBS_O_WORKDIR: executa o job a partir do diretório onde o comando qsub foi chamado.
  • conda activate meu_env: ativa o ambiente com os programas necessários.
  • python script.py: representa o comando principal da análise.

Após salvar o script, submeta o job com:

qsub script.pbs

Organização de dados

Organize scripts e dados em diretórios separados. Isso facilita a reprodutibilidade, evita perda de arquivos e reduz duplicações.

/home/usuario/
/data/usuario/projeto/
  • /home/usuario/: use para scripts, arquivos pequenos de configuração e documentos de trabalho.
  • /data/usuario/projeto/: use para dados brutos, resultados intermediários e saídas de análises.

Sempre que possível, crie uma pasta por projeto e use nomes claros para scripts, amostras e resultados.

Armazenamento e limpeza

  • /home: scripts.
  • /data: dados.
  • Remova arquivos temporários e evite duplicações.

Instalação de softwares e bases de dados

  • É proibida a instalação global sem autorização.
  • É proibido usar sudo.
  • É proibido baixar grandes bases em diretórios pessoais.
  • Prefira ambientes conda/mamba e documente versões.
  • Solicitações devem ser feitas ao ponto focal do grupo.

Bases de dados devem ser centralizadas, evitando duplicação.

Reprodutibilidade, backup e conduta

  • Versione scripts com Git, registre parâmetros e garanta rastreabilidade.
  • O servidor não é sistema de backup. Usuários são responsáveis pelos seus dados.
  • Comunique uso intensivo, respeite períodos de manutenção e priorize o uso coletivo.

Proibições

  • Mineração de criptomoedas.
  • Uso ilegal de software.
  • Execução de código malicioso.
  • Sobrecarga intencional.

Infraestrutura

O ambiente computacional do servidor LOTR é baseado em um Dell PowerEdge R7625, Rack 2U, Dual Socket, com arquitetura de alto desempenho adequada para análises bioinformáticas intensivas.

Processamento

2x AMD EPYC 9354, 64 núcleos no total e alta capacidade de paralelização para pipelines de bioinformática.

Memória RAM

512 GB DDR5 ECC, com suporte a grandes volumes de dados.

Armazenamento

Aproximadamente 15 TB NVMe de alta performance, otimizado para leitura intensiva de dados.

Rede

Interfaces de 10 GbE e 1 GbE para alta taxa de transferência de dados.

GPU

Equipado com 2 GPUs NVIDIA L4 (24 GB cada) para computação acelerada, modelagem, filogeografia, IA etc.

Gerenciamento

Monitoramento remoto via iDRAC, redundância de energia e resfriamento.
